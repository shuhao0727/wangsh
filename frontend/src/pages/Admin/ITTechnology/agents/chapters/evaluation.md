# 评估体系：给 AI 出张试卷

## 你有没有想过...

每次大考完，你都会拿到一个分数——语文 85、数学 92、英语 78。但你想过没有：**谁来给 AI 打分？怎么打？**

这个问题比你想象的难得多。给你出一道"3 乘 5 等于几"——答案只有一个，对就是对，错就是错。但如果问题是"帮我设计一个好看的个人主页"呢？100 个设计师能给出 100 种方案，每个都"不错"，但也每个都"不够完美"。你怎么判断 AI 的作品是 85 分还是 92 分？

这就是 AI 评估（Evaluation）要解决的问题。**评估 Agent 就像批改试卷——不是只看最后答案对不对，还要看解题步骤好不好。**

---

## 为什么考 AI 比考你更难？

想象你在王者荣耀里打排位。输赢很明确——基地炸了就是输了。但如果要评价一个辅助玩家呢？他可能 KDA 是 0/5/18（击杀 0、死亡 5、助攻 18）。光看击杀数——太菜了！但看助攻——全场最高！那这个辅助到底打得好不好？

AI 评估面临同样的困境：

**困境一：AI 像即兴表演，不是计算器。** 同样的问题问两次，AI 可能给出不同的答案。你不能用"答对一次就过关"来测试它——需要测很多次，看"平均表现"。

**困境二：没有标准答案。** 让 AI 总结一篇文章——什么叫"好的总结"？简洁算好还是详细算好？不同人有不同标准。

**困境三：错误会传染。** AI 做任务像走迷宫，第一步拐错弯，后面全跑偏。最终失败的原因可能藏在第一步，很难找到。

**困境四：AI 依赖外部世界。** 它要搜网页、调 API。如果搜索网站挂了，AI 也无能为力。但这是 AI 的错吗？

## 🎮 类比理解

**王者荣耀段位系统**就是一套完整的"评估体系"：

- 赢了多少场 → 任务成功率，相当于"做对了几道题"
- KDA（击杀/死亡/助攻比）→ 过程质量，相当于"解题步骤好不好"
- MVP 次数 → 工具使用能力，相当于"会不会在关键时刻用闪现"
- 段位分（星耀/王者/荣耀）→ 综合评分

但游戏还有一个 AI 评估没有的优势：**每一局的"回放"都是完整的。** 赢了你能看回放学操作，输了你能看回放找失误。AI 评估也要做到这一点——不光看"赢了没"，还要看"每一波团战怎么打的"。

再换一个角度，用**原神的深境螺旋**来理解：

深境螺旋每一层有时间限制，打得太慢就拿不到三星。这跟评估 AI 的"效率"指标一模一样——不是只看"打没打过"，还要看"用了多少时间"、"死了几次"。一个玩家用 1 分钟满血通关（效率高），另一个用 5 分钟残血通关（效率低），虽然结果都是"通关"，但水平完全不同，对吧？

## 评估 AI 的三层体系

**第一层：任务成功率——"这道题做对了吗？"**

最直观的指标。给 AI 100 道题，看它做对多少道。

```
任务成功率 = 做对的题数 / 总题数
```

但"做对"的定义需要细化。有三种判分方式：

- **二元判定（对就是对，错就是错）**：适合有明确答案的任务。像选择题——选 C 就得分，选别的就错了。
- **部分评分（步骤分）**：像数学大题——结果错了不一定零分，过程对了也能拿分。公式写对了 +2 分，代入正确 +2 分，结果算错扣 1 分。
- **自动验证**：写代码的任务直接跑测试用例，通过了就是对。

**第二层：过程质量——"你是怎么做的？"**

两个同学都做对了一道数学题：小明用巧妙公式两步搞定，小红绕了十个弯才做出来。虽然结果一样，但小明的"解题质量"明显更高。AI 也一样，要分析它的**轨迹（Trajectory）**——每一步的思考是否合理、有没有走弯路、遇到错误会不会自我纠正。

**第三层：工具使用能力——"会用工具吗？"**

AI Agent 的核心能力不是"聪明"，而是"知道什么时候该用什么工具"。用户问"今天天气怎么样"，正确的做法是调用天气 API，而不是打开计算器。

## 动手试一试：AI 考试模拟器

```python
# 给 AI 出卷子：模拟评估三个版本的 AI
import random

class AIStudent:
    """一个等着被考试的 AI"""
    def __init__(self, name, 正确率, 效率, 工具水平):
        self.name = name
        self.正确率 = 正确率
        self.效率 = 效率
        self.工具水平 = 工具水平

    def do_exam(self, 题数):
        """做一套卷子"""
        score = 0
        for i in range(题数):
            if random.random() < self.正确率:
                # 模拟部分评分：不是全对全错
                step_bonus = random.uniform(0, self.效率 * 30)
                tool_bonus = 10 if random.random() < self.工具水平 else 0
                score += 60 + step_bonus + tool_bonus  # 满分 100
        return score / 题数

# 创建三个考生
v1 = AIStudent("学习助手 V1.0", 0.65, 0.5, 0.6)
v2 = AIStudent("学习助手 V2.0", 0.78, 0.7, 0.75)
v3 = AIStudent("学习助手 V3.0", 0.92, 0.9, 0.85)

# 考试！
for ai in [v1, v2, v3]:
    avg = ai.do_exam(50)
    grade = "⭐⭐⭐⭐⭐ S级" if avg >= 85 else ("⭐⭐⭐⭐ A级" if avg >= 70 else "⭐⭐ B级")
    print(f"{ai.name}：平均 {avg:.1f} 分 → {grade}")

print("\n💡 注意：真实评估要测上万道题，不能只做 50 道！")
```

## 让用户来"投票"：A/B 测试

还有一种评估方法很直观：**让真实用户来选择。**

假设你有两个版本的 AI 学习助手——A 版是"话唠型"（详细讲解每一步），B 版是"简洁型"（直接给答案）。你不知道哪个更好，那就同时上线两个版本，各分给一半用户。一周后看数据：哪个版本的用户满意度更高？哪个版本的问题解决率更高？

这就是**A/B 测试**——AI 界的"大众点评"。就像学校食堂推出两道新菜，哪边排队更长，哪道菜更受欢迎。

反馈发现：备考前临时抱佛脚时，同学更喜欢简洁型（快！）；平时慢慢学的时候，话唠型更好（详细！）。同一个 AI，不同场景下评估结果完全不同——这就是为什么评估体系需要"看情况"。
