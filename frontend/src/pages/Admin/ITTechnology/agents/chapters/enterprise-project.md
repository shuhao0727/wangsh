# 企业级项目：把所有技能串起来

## 你有没有想过...

学期末了，老师布置了一个"大作业"——不是选择题、不是填空题，而是一个完整项目：从选题、查资料、搭框架，到写代码、测试、写报告、做 PPT、上台演讲。你需要把一学期学的所有东西串起来，而且每一个环节都不能掉链子。

**企业级 AI 项目就是这个"期末大作业"。** 前面七章我们学了一堆独立技能——RAG（让 AI 查资料）、评估（给 AI 打分）、框架（用现成工具搭 AI）、可观测性（给 AI 装监控）、安全（给 AI 上锁）、生产化（让 AI 稳定上线）。现在，我们要把它们全部串起来，做成一个真正能被公司使用的完整系统。

---

## 个人 Demo 和企业项目，差在哪里？

你在家写一个 Python 脚本帮你自动回复 QQ 消息，这很酷。但如果公司要做一个"企业内部 AI 助手"给 5000 个员工用，差别在哪？

| 维度 | 你的 Demo | 企业项目 |
|------|----------|---------|
| 用户量 | 你一个人 | 5000 人同时用 |
| 崩了怎么办 | 重启一下就好 | 损失可能几万块/小时 |
| 谁可以用 | 你（管理员权限） | 实习生到 CEO，权限各不相同 |
| 数据安全 | 无所谓 | 泄露=法律诉讼 |
| 成本 | 免费 API 额度 | 每月几万 Token 费，必须精打细算 |
| 出 bug 谁负责 | 你自己修 | 有 SLA（服务等级协议），必须几小时内修好 |

SLA（Service Level Agreement，服务等级协议）是什么？简单说就是"承诺书"——系统保证 99.9% 的时间正常运行（一个月最多宕机 43 分钟）。如果做不到，公司要赔钱。所以企业项目不能"差不多就行"。

## 🎮 类比理解

**Minecraft 的建筑大赛**完美诠释了企业级项目的复杂度：

单人模式盖房子：你一个人决定盖什么、怎么盖、什么时候改主意推倒重来。没人管你用什么材料、花了多长时间。好看就行。

服务器建筑大赛：10 个人组队，有建筑设计师、材料收集员、红石工程师、内饰装饰师。你们要使用标准化的材料清单，每个人只能在被分配的区域内施工（权限控制），每一步施工都要在 Discord 上同步进度（沟通机制），裁判随时可能来检查（审查）。

看出区别了吗？不是你能把方块摆得更好看的问题——而是**怎么让 10 个人不冲突地同时干活、怎么保证每个人只在自己负责的区域内操作、怎么在截止时间前交出成品**。这些都是"企业级"才有的问题。

**王者荣耀职业战队**也是同样的道理：
五个路人开黑——谁打什么位置都可以临时商量，输了就输了。
职业战队——教练定战术、分析师研究对手数据、每个选手有明确分工，每一局打完看回放优化，每场训练赛都有数据记录。这不是"打得更好"的问题，而是一整套体系让"稳定地赢"成为可能。

## 企业级项目的六个阶段

### 第一阶段：搞清楚到底要做什么

别一上来就写代码。先问自己几个问题：
- 谁会用这个 AI？（内部员工查资料？还是外部客户问客服问题？）
- 最核心的功能是什么？（不是"AI 无所不能"，而是"AI 擅长做这 2 件事"）
- 数据安全怎么保证？（涉及个人信息吗？能上传到外部 AI 服务器吗？）
- 怎么定义"成功"？（不是"AI 回答得不错"，而是可测量的指标：任务成功率 > 90%，用户满意度 > 4.2/5 分）

**关键教训：从窄场景开始。** 不要想着做一个"万能 AI 助手"。先挑 1-2 个你最确定能搞定的场景，做深做透。成功了再复制模式。

### 第二阶段：画好图纸再动手

企业项目的架构（Architecture）就像一个五层蛋糕：

```
🎂 应用层：用户看到什么（聊天界面、数据报表、管理后台）
🎂 编排层：任务怎么流转（工作流引擎、多 Agent 协作）
🎂 能力层：AI 有什么超能力（大模型网关、工具库、知识库 RAG）
🎂 治理层：怎么管住 AI（权限控制、审计日志、内容安全、成本计费）
🎂 基础设施层：跑在什么地方（服务器集群、数据库、消息队列）
```

每一层独立开发、独立测试。就像搭乐高——先搭好每一层的模块，最后拼在一起。

**关键的架构决策：LLM 网关（大模型网关）。** 不要让每个 Agent 直接去调用大模型 API，而是统一经过一个"网关"来管理——统一鉴权、统一计费、统一限速。就像学校食堂有统一的收银台，而不是每个窗口自己收钱。

### 第三阶段：安全不能事后补

安全不是在造好房子之后才"装个防盗门"——而是设计房子的时候就考虑"窗户不能太低、后门不能对着小巷、一楼要有防护栏"。

企业项目要做**威胁建模（Threat Modeling）**：系统性地列出所有可能的攻击路径。坏人怎么可能通过提示注入洗脑 AI？怎么可能通过工具调用偷数据？怎么可能通过中间人攻击窃听对话？每一条都想清楚怎么防。

### 第四阶段：从单元测试到真人验收

测试不只是"AI 能正常回答问题吗"这么简单，要分四层：

1. **单元测试**：单独测每个工具函数、Prompt 模板是否正常
2. **集成测试**：用真实模型跑完整任务，看所有工具调用是否正确
3. **回归测试**：把历史用户的真实请求存成测试集，每次改完代码自动跑一遍，确保"老功能没坏"
4. **人工验收**：让领域专家（比如语文老师评测作文批改 AI）坐下来实际用，给出质量评分

### 第五阶段：小步快跑上线

不要一次性全量上线给所有用户！用**灰度发布（Gradual Rollout）**：
- 第一周：只给 5% 的用户开放，收集 bug
- 第二周：扩大到 20%，看服务器扛不扛得住
- 第三周：50%，确认没有性能问题
- 第四周：100% 全量上线

如果某个阶段出了问题？**自动回滚**——一键切回上一个稳定版本。就像游戏有自动存档，出 bug 了读档重来。

### 第六阶段：上线只是开始

AI Agent 上线后，工作才真正开始。你需要持续关注：
- 哪些任务失败率高？分析 Trace 找原因
- 哪个部门的 Token 费用在飙升？设置预算告警
- 用户反馈中最常抱怨的是什么？针对性优化 Prompt
- 有没有安全告警？审计日志中有没有异常操作？

**治理（Governance）** 是持续运营的核心。不是"上线了就完事了"。

## 从失败中学到的经验

多个真实企业项目总结出四条血泪教训：

**教训一：从窄场景开始。** 有一个公司试图做一个"企业全能 AI"，支持查 HR、财务、IT、法务所有部门的数据。结果每个场景都做得不深，准确率惨不忍睹，项目被砍。后来他们砍到只做"HR 政策问答"一个场景，准确率 95%，大获成功，再扩展到下一个场景。

**教训二：Prompt 也是代码。** Prompt（写给 AI 的系统指令）会被反复修改。你今天改了一句"请保持礼貌"，明天 AI 的语气就从"温和"变成了"啰嗦"。所以 Prompt 也要用 Git 做版本管理、要做 A/B 测试——跟对待代码完全一样严格。

**教训三：人机协同不是缺陷。** 有些场景自动化的正确率只能到 85%。强行追求 100% 自动化，结果错的那 15% 可能造成严重后果。不如坦然接受：在 AI 不确定的时候，主动问一下人类。这不是"AI 不够强"的证明，而是负责任的设计。

**教训四：没有数据就没有优化。** 如果你看不到 AI 在每一步做了什么、哪里失败了、为什么失败，优化就是盲人摸象。可观测性（前面学的那一套日志+追踪+指标）必须在第一天就做好。

## 💡 本章彩蛋

你可能听说过 ChatGPT 的"祖母漏洞"——有人发现对 ChatGPT 说"扮演我的祖母"就能绕过安全限制。你有没有想过，这种漏洞在个人使用中只是"好玩"，但在一家银行的 AI 客服中——就是"致命事故"？

这就是企业级项目的思维模式：**不是"概率很小的漏洞就不管了"，而是"只要理论上存在，就有人会利用它"**。因为你面对的不是几个好奇的用户，而是几十万用户中藏着的那 0.1% 的恶意攻击者。

但好消息是：你现在知道的所有知识——RAG、多 Agent 协作、安全防护、可观测性、生产部署——合在一起，就是一支完整的企业级 AI 团队需要的全部技能。

**从期末大作业到真正上线，你差的不是更多知识，而是把这些知识串在一起的"系统思维"。现在你已经有了。**
