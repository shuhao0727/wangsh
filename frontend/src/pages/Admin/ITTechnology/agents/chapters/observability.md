# 可观测性：AI 的"战斗回放"

## 你有没有想过...

你在王者荣耀里打完一局排位，输了。你很郁闷，但你点了"保存回放"。打开录像一看：原来第 8 分钟你一个人去对面野区反野，队友发信号你都没看到，被三个人蹲了。就是那一波团灭，被一波推了水晶。

没有回放的话，你只知道"输了"，不知道"输在哪里"。

AI Agent 也是一样的。如果 AI 的任务成功率是 85%，你只知道"15% 的情况下它会失败"。但你不知道这 15% 是怎么失败的——是在哪一步出错的？是模型的问题还是工具的问题？为什么上午表现好、下午表现差？

**可观测性（Observability）就像游戏的战斗回放——每一步都能回看，找到哪里出了问题，下次不再犯同样的错误。**

---

## 三大武器：日志、追踪、指标

可观测性有三样核心武器，各自干不同的事：

**日志（Logs）= 日记本**
AI 每做一件事，都在"日记本"里记一笔。什么时候调用了模型、用了多少 Token（AI 的"算力花费"）、调用了什么工具、结果成功还是失败。每条日志都是"某某时间，做了某某事"。

**追踪（Tracing）= 运动轨迹**
日志是散的，追踪把它们用一根线串起来。同一个任务的每一步都打上相同的 Trace ID，你就能看到完整的执行路径："先规划（用了 2 秒）→ 搜索网页（用了 0.3 秒）→ 分析结果（用了 1.5 秒）→ 生成回答（用了 3 秒）"。就像跑步 App 记录你跑了多少公里、每公里配速是多少。

**指标（Metrics）= 体检数据**
日志和追踪是细节，指标是汇总。你不看每天的血糖值，而是看"最近一周血糖平均值"。AI 的指标也一样：过去 24 小时平均响应时间、Token 总消耗、工具调用成功率。

---

## 🎮 类比理解

**Minecraft 的死亡画面**是你最熟悉的"可观测性系统"：

你掉进岩浆死了，屏幕上显示："你死于火焰，死亡时坐标为 X=120, Y=64, Z=-300，背包里有 37 个铁锭、2 个钻石..."——这不就是 AI 异常事件的日志吗？每个细节都记下来了，方便你回去捡装备（或者复盘"我为啥要站在岩浆旁边"）。

**原神的深境螺旋结算画面**就是指标仪表盘：通关时间、受击次数、最高伤害数值——汇总成一行行数据。你不需要看完整的"操作回放"就知道这层打得怎么样。

**王者荣耀的"观战模式"**就是最完整的 Trace 系统：不仅能看你自己，还能看你队友、对面五个人的全部操作。你能看到每一步走位、每个技能释放时机、每个装备购买时间点。这就是为什么职业选手能靠看回放变强——可观测性的终极目标就是让"变强"成为可能。

## 一个完整 Trace 长什么样？

假设 AI 接到一个任务："帮我查一下今天北京的天气，推荐合适的穿搭"。

```
任务 Trace (ID: abc-123)
│
├── Step 1 [LLM调用] 规划阶段
│   模型：claude-sonnet
│   Token：1200 个
│   耗时：2.3 秒
│   状态：✅ 成功
│
├── Step 2 [工具调用] 查询天气 API
│   工具：weather_api
│   参数：{"city": "北京"}
│   返回：25°C，晴天
│   耗时：0.35 秒
│   状态：✅ 成功
│
├── Step 3 [LLM调用] 分析天气数据
│   Token：800 个
│   耗时：1.5 秒
│   状态：✅ 成功
│
├── Step 4 [工具调用] 搜索穿搭建议
│   工具：web_search
│   参数：{"q": "25度 穿搭 男生"}
│   耗时：0.28 秒
│   状态：✅ 成功
│
└── Step 5 [LLM调用] 生成最终回答
    Token：600 个
    耗时：3.1 秒
    状态：✅ 成功

━━━━━━━━━━━━━━━━━━━━
总计：LLM 调用 3 次 | 工具调用 2 次 | 总耗时 ~7.5s | Token 消耗 2600
```

现在，如果另一个 Trace 显示 Step 4 的 web_search 工具失败了（超时），AI 该怎么做？好的 AI 会说"搜索暂时不可用，我先用已有知识回答"然后继续。差的 AI 会一直重试直到系统崩溃。

**没有 Trace 的话，你只会看到"这个任务的用户不满意"——但你永远不会知道是搜索工具超时导致的。**

---

## 出 bug 了怎么办？调试三招

**招式一：回放模式（确定性回放）**
把整个任务的所有条件原封不动地记录下来，在调试环境里一模一样重演一遍。像游戏存档——你可以在 boss 战前存档，打输了就读档重来，反复观察是哪个操作出了问题。

**招式二：逐帧检查（单步调试）**
开发者手动推进 AI 的每一步，像暂停视频一样在每一帧停下来看：AI 在这一步想了什么？这个想法合理吗？如果中间修改一个参数会怎样？

**招式三：对比分析**
拿一个成功的任务和一个失败的任务，把它们的 Trace 并排对比。在哪一步开始分叉的？分叉点就是问题所在。

## 动手试一试：迷你追踪器

```python
# 给 AI 装上"行车记录仪"——迷你追踪系统
import time
import uuid

class MiniTracer:
    def __init__(self):
        self.traces = {}

    def start(self, task_name):
        """开始追踪一个新任务"""
        tid = str(uuid.uuid4())[:6]
        self.traces[tid] = {
            "task": task_name,
            "steps": [],
            "start": time.time()
        }
        print(f"🔍 开始追踪：「{task_name}」 (ID: {tid})")
        return tid

    def step(self, tid, name, success, duration):
        """记录一步操作"""
        self.traces[tid]["steps"].append({
            "name": name,
            "success": success,
            "duration": duration
        })
        icon = "✅" if success else "❌"
        print(f"  {icon} {name} ({duration}ms)")

    def report(self, tid):
        """打印完整报告"""
        t = self.traces[tid]
        total = sum(s["duration"] for s in t["steps"])
        fails = sum(1 for s in t["steps"] if not s["success"])
        print(f"\n📊 任务「{t['task']}」报告：")
        print(f"  总步数：{len(t['steps'])} | 总耗时：{total}ms | 失败步骤：{fails}")
        if fails > 0:
            for i, s in enumerate(t["steps"]):
                if not s["success"]:
                    print(f"  ⚠️ 第 {i+1} 步「{s['name']}」失败了！")

# 测试
tracer = MiniTracer()
tid = tracer.start("查询天气并推荐穿搭")
tracer.step(tid, "LLM规划", True, 2100)
tracer.step(tid, "查天气API", True, 350)
tracer.step(tid, "分析数据", True, 1500)
tracer.step(tid, "搜索穿搭", False, 5000)  # 超时了！
tracer.step(tid, "降级处理（手动推荐）", True, 2000)
tracer.report(tid)

print("\n💡 第 4 步「搜索穿搭」失败了——如果只看最终成功率 80%，")
print("   你永远不知道问题出在搜索工具上，而不是 AI 本身！")
```

## 💡 本章彩蛋

你有没有注意过——职业电竞选手打比赛的时候，屏幕上永远有一个"教练视角"的小地图，能看到所有人的位置。而普通玩家只能看到自己周围的视野。这个差别，就是"有可观测性"和"没有可观测性"的差别。

普通开发者部署 AI 后，能看到的就是"任务成功率 85%"这一个数字——相当于你打游戏只看到一个"赢了"或"输了"。专业团队部署 AI，就像职业选手——大屏幕上实时显示着几十个指标：响应时间、Token 消耗、工具成功率、错误类型分布...

**下次你 AI 出 bug 的时候，别只会说"它又不行了"。装一个追踪器，看看它到底在哪一步翻的车。记住：看不到问题，就修不了问题。**
